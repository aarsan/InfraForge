Service Onboarding Pipeline

A complete guide to how InfraForge generates, validates, deploys, and approves Azure infrastructure — a 12-step pipeline with AI-powered auto-healing (5 attempts per cycle, up to 2 regeneration cycles).

When you click "Start Onboarding" on a service, InfraForge runs a 12-step pipeline that takes a service type (e.g. Microsoft.Network/virtualNetworks) and produces a fully validated, deployed-and-cleaned-up ARM template ready for production use. Every step is streamed to the UI in real time as NDJSON events.

Model Routing Table

InfraForge uses task-based model routing — each pipeline step is handled by the LLM best suited for that specific type of work.

TaskModelWhy This Model
PLANNING claude-sonnet-4 Architecture planning + root-cause analysis — consistent with generation and fixing models
CODE_GENERATION claude-sonnet-4 Precise, structured ARM/Bicep code generation with strong instruction adherence
CODE_FIXING claude-sonnet-4 Template healing needs strong instruction adherence to apply policy-required changes
POLICY_GENERATION claude-sonnet-4 Policy JSON structure + logical condition reasoning
VALIDATION_ANALYSIS claude-sonnet-4 Analyzing What-If results, deploy errors, and policy violations — consistent with planning model
GOVERNANCE_REVIEW claude-sonnet-4 CISO/CTO review gate — deep reasoning about security, architecture, and compliance
QUICK_CLASSIFY gpt-4.1-nano Simple classification, routing, or yes/no decisions — speed wins
DESIGN_DOCUMENT gpt-4.1 Long-form technical writing with structured output

Pre-Loop Steps (Steps 1–7, Run Once)

1 Model Routing Initialization

Builds the per-task model routing table, cleans up stale draft versions, and emits each assignment to the UI. The user sees which LLM handles each step — full transparency. Also emits an overview of all 12 pipeline steps.

No LLM call Cannot fail

2 Dependency Gates

Checks whether the service has required dependencies that must be onboarded first (e.g. a subnet requires a virtual network). If any dependencies are missing or not yet fully onboarded, the pipeline auto-onboards them inline by spawning recursive sub-pipelines.

A circular dependency guard prevents infinite loops — if service A depends on B which depends on A, the cycle is detected and the gate proceeds.

No LLM call May spawn sub-pipelines

3 Organization Standards Analysis

Fetches all applicable organization standards (security, tagging, networking, compliance) from the governance database. Builds three separate context bundles used by later steps: standards_ctx (for generation), policy_standards_ctx (for policy gen), and governance_ctx (for governance review).

Each standard is displayed with severity: 🔴 Critical 🟠 High 🟡 Medium 🟢 Low

No LLM call

4 Architecture Planning claude-sonnet-4

The model analyzes the service type, org standards, and security requirements to produce an architecture plan. This plan specifies:

  • Resources to create (type, API version, purpose)
  • Security configs (TLS, encryption, managed identity)
  • Parameters to expose with default values
  • Critical properties for production readiness
  • Standards compliance mapping
  • Validation criteria for later verification
The plan is fed into all subsequent LLM calls — the code generation model follows the plan precisely rather than reasoning about architecture itself.
5 ARM Template Generation claude-sonnet-4

The code generation model produces an ARM template guided by the architecture plan + org standards.

After generation, post-processing runs: sanitize resource names, inject required tags, strip foreign properties, stamp metadata, and a safety guard (_ensure_parameter_defaults) injects missing defaultValue on every parameter.

Fatal if: Generation returns empty content or invalid JSON. Pipeline aborts — no healing possible at this stage.
6 Azure Policy Generation claude-sonnet-4

Generates an Azure Policy definition that enforces organizational standards on the deployed resources. Used in Step 8.6 (Runtime Policy Test) to verify the actual deployed resources comply with org requirements.

Non-fatal If LLM generation fails, a deterministic policy fallback is generated from rules instead. The runtime policy test is only skipped if both the LLM and the fallback produce nothing.

7 Governance Review (CISO + CTO Gate) claude-sonnet-4

A structured security and architecture review gate. Two parallel reviewer personas (CISO and CTO) evaluate the generated ARM template against organizational security requirements and architectural best practices.

If the CISO review identifies blocking security issues, the template enters its own governance heal loop (up to 5 attempts) — the template is repaired to address security concerns, then re-reviewed. If all 5 governance heal attempts are exhausted, the pipeline proceeds to the main healing loop anyway (as a last resort).

CISO blocks → Governance heal loop (×5).
Both approve → Proceed to healing loop.

Step 8: Healing Loop (Up to 5 Attempts × 2 Regen Cycles)

How it works: Each attempt runs sub-steps 8.1→8.6 sequentially. If any sub-step fails and it's not the last attempt, the template is sent to claude-sonnet-4 for repair. The fixed template is re-validated from the beginning. A heal history tracks every fix attempt so the LLM never repeats the same failed approach.

Regeneration: If all 5 heal attempts are exhausted, the pipeline can trigger a full regeneration cycle — re-planning architecture with error context, then generating an entirely new ARM template. Up to 2 regeneration cycles are allowed, each resetting the heal counter.
8.1 Parse JSON

Validates the ARM template is syntactically valid JSON via json.loads(). Reports line number and column on failure. Also runs validate_arm_references() to auto-fix missing variables and parameters before proceeding.

On failure → Auto-heal with error context, then restart loop.

8.2 Static Policy Check

Evaluates the template against all organization governance policies before touching Azure. Checks include required tags, allowed regions, security settings, and custom org rules. Each check emits a pass/warn/fail result.

On blockers → Builds a targeted fix prompt listing each failed check. Auto-heals and restarts loop.

8.3 ARM What-If Azure RM API

Submits the template to Azure Resource Manager's What-If endpoint — like terraform plan but native. Previews what resources would be created, modified, or deleted without actually provisioning anything.

Transient errors (throttled, resource group being deleted, service unavailable) do not burn a heal attempt — the pipeline waits 10 seconds and retries.

On template error → Auto-heal with ARM error message, restart loop.

8.4 ARM Deployment Azure RM API

Actually provisions resources into a temporary validation resource group. Deployment mode is incremental (doesn't delete existing resources). If the error message is generic, InfraForge fetches operation-level errors for better diagnostics. A heartbeat event is emitted every 20 seconds during long deployments to keep the HTTP connection alive.

On deploy error → Auto-heal. On last attempt (no regen cycles left), cleans up the resource group then fails the pipeline.

On success → Resource group is marked for cleanup in Step 11.

8.5 Resource Verification Azure RM API

Lists all resources deployed into the validation resource group and fetches their full properties (tags, network config, security settings). These properties are needed for the runtime policy evaluation in the next step.

Non-fatal If enumeration fails, the policy test proceeds with an empty resource list.

8.6 Runtime Policy Compliance Test

Evaluates each deployed resource against the generated Azure Policy using a local policy engine. This engine interprets the policy's if condition tree (allOf / anyOf / not) with operators like equals, notEquals, in, contains, exists, greater, less.

Nested policy healing: If resources violate the policy, this step first attempts to relax the policy (up to 3 times) to match the deployed resources before falling back to healing the ARM template. This avoids unnecessary template churn when the policy is overly restrictive.

On violation (after policy heal) → Auto-heal template with violation details + policy content. On last attempt, cleans up RG then fails (or triggers regen).

Skipped if no policy was generated or no resources exist.

Post-Loop Steps (Steps 9–12, On Success)

9 Infrastructure Testing claude-sonnet-4

AI-generates infrastructure smoke tests for each resource type in the deployed template. Tests validate real resource behavior: connectivity, configuration, security settings. Test results are captured in the validation record.

Advisory by default — test failures are reported but don't block the pipeline. Set INFRA_TEST_BLOCKING=true to make failures block.

10 Deploy Azure Policy Azure RM API

Deploys the generated (or healed) Azure Policy definition to Azure, making it available for ongoing compliance enforcement across the organization.

Non-blocking — if deployment fails, the error is logged but does not fail the pipeline. Policy cleanup on failure is automatic.

11 Cleanup Validation Resource Group

Deletes the temporary validation resource group and any deployed policy artifacts. This is fire-and-forget — the pipeline doesn't wait for full deletion (which can take minutes). A safety-net finally block ensures cleanup even if an unhandled error occurs.

12 Promote to Approved + Co-onboard ✅

Builds a comprehensive validation result capturing What-If results, deployed resources, static policy check, runtime policy compliance, test results, and attempt count. Updates the service version status to "approved" and sets it as the active version.

After promotion, the pipeline checks for always_include child resources (e.g. subnets for a virtual network) and auto-onboards them by spawning additional onboarding sub-pipelines. This ensures composite resource types are fully onboarded together.

🎉 Service onboarded! The ARM template is now marked as production-ready and available in the service catalog. Any child resources are being co-onboarded automatically.

Healing Loop Mechanics

How Auto-Healing Works

When a sub-step fails, the current template + error message + full heal history is sent to claude-sonnet-4 (the CODE_FIXING model) via a two-phase process: first an analysis call identifies the root cause, then a repair call produces the corrected template. The healed template then goes through the full validation loop again from step 8.1.

Escalation Strategy

AttemptStrategyPrompt Additions
1 Standard fix Error message + critical rules (keep location expressions, keep resource intent)
2–3 Fundamentally different approach "Try a different API version, restructure dependencies, remove problematic sub-resources"
4–5 Drastic simplification "Strip to primary resource only, remove diagnosticSettings/locks/autoscale, use simplest config"

Heal History Tracking

Each heal attempt is recorded as {attempt, phase, error, fix_summary}. The summarize_fix() function produces a structural diff (resource count changes, API version changes, added/removed resources, parameter changes). The full history is included in subsequent fix prompts with: "DO NOT repeat these fixes — they already failed."

Post-Heal Safety Guards

  1. Empty response → Keep original template
  2. Non-JSON response → Try to extract JSON object, else keep original
  3. Invalid JSON → Keep original template
  4. Location corruption → Restore [resourceGroup().location]
  5. Missing defaults → Inject via _ensure_parameter_defaults()
  6. Markdown fences → Strip automatically

Regeneration Cycles

If all 5 heal attempts in a cycle are exhausted without success, the pipeline doesn't immediately fail. Instead, it can trigger a full regeneration cycle (up to 2 times):

  1. Re-plan — Calls the PLANNING model again with full error context from the failed cycle. The prompt includes every error encountered and explicitly asks for a fundamentally different architectural approach.
  2. Re-generate — Produces an entirely new ARM template from the new plan.
  3. Reset — Clears the heal history and resets the attempt counter to 1.
  4. Re-enter loop — The new template goes through steps 8.1→8.6 with a fresh set of 5 heal attempts.
Total resilience: 5 heal attempts × (1 initial + 2 regen cycles) = up to 15 template fix attempts before the pipeline gives up. Combined with the governance heal loop (×5) and the policy heal sub-loop (×3 per attempt), the pipeline has substantial self-repair capacity.

Transient Error Handling

Certain Azure errors are not template problems — they're infrastructure timing issues. These are detected by keyword matching and handled specially:

Keyword PatternMeaningAction
beingdeleted / being deletedRG from previous run still deletingWait 10s, retry
deprovisioningResources still deprovisioningWait 10s, retry
throttled / toomanyrequestsARM rate limitingWait 10s, retry
serviceunavailableAzure service outageWait 10s, retry
internalservererrorAzure server errorWait 10s, retry
Key detail: Transient retries do not burn a heal attempt. The pipeline can retry indefinitely for transient issues while preserving all 5 heal attempts for actual template problems.

Parameter Injection

ARM templates are deployed with explicit parameter values extracted from the template's defaultValue fields. This prevents the common failure:

The value for the template parameter 'resourceName' at line 1 and column 184 is not provided.

Two layers of protection:

  1. _ensure_parameter_defaults() — Runs after generation AND after every heal. Injects defaultValue into any parameter missing one.
  2. _extract_param_values() — Before each deploy/what-if call, extracts default values from the template and passes them as explicit parameters={...}. Skips ARM expressions like [resourceGroup().location].
